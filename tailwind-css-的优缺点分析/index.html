1.前言

tailwind css是目前社区非常流行的原子化css方案，所谓原子化css是一种用于设计和构建现代 Web 应用程序的 css 框架，它提出了一种将 css 拆分为更小、更可维护的部分的方法，以便更好地组织和管理css代码

在原子化 css 中，css 组件被拆分为更小的部分，这些部分可以独立地编辑、测试和重用。这些原子通常是单个像素或极其微小的变化，例如颜色、大小、位置等。这些原子可以组合成更复杂的组件，例如按钮、表单、图标等。原子化 css 有助于减少代码量，提高代码的可维护性和可重用性，同时还有助于提高开发效率

比如当给一个标题设置样式时，我们希望只写它的类名就加载所需样式，而不是单独还要定义一套css

2.tailwind css的配置

2.1.安装基础的依赖库

以next.js为例，首选需要安装几个基础依赖npm install -D tailwindcss postcss autoprefixer，因为tailwind css是需要postcss为基础，以插件的形式接入到项目中

1

npm install -D tailwindcss postcss autoprefixer

2.2.初始化项目

1

npx tailwindcss init -p

2.4.将tailwind css配置引入全局

在项目中的globals.css里，添加@tailwind指令

3.tailwind css具备优势

随着越来越多的团队选择tailwind css，说明它总是有过人之处的

3.1.有效减少css体积

css文件的下载和解析虽然不会阻塞DOM的解析，但是却会阻塞页面的渲染，这实则上是浏览器为了优化重绘和回流的频率所做的优化机制。由此，我们可以很清晰的知道，css的文件体积是性能优化的一个关键问题。项目打包出来的css文件或者代码，往往会伴随着页面的增加而变大，进而最终影响页面的性能，其中最为直观的指标就是首屏渲染时间

而针对这个问题，自然有一些编译工具提供的css 代码分割、压缩等功能。但最有效的方法其实就是提高样式代码的复用性，以及在编译层面上尽可能的删除掉不需要的css代码，从而让css文件体积变得更小

比如以下两个class

.containerC {
  position: relative;
  width: 100px;

随着项目的增大、开发人员越来越多，有些时候有的人为了图方便省事甚至直接随意定义一个类名，自己用的爽就行了。这个时候，项目中的position: relative、 width: 100px就会越来越重复，导致css文件体积的不断增大

而使用tailwind之后，全局会定义一个relative class name，而width则是动态的通过插件编译生成的w-100，不管怎么说，只会有一个 position: relative、width: 100px，这就极大的减少了重复代码，减少了css的体积

3.2.css的模块化

为了避免全局的样式污染，之前社区已经提供了多种方案，比如css in js、css modules，但是仍然还有缺陷

css modules：通过计算hash值将书写的css类名更改成全局唯一的值，虽然有效的解决了样式污染问题，但是需要使用全局样式时，需要额外的定义:global的y样式，以及每个模块都需要单独的引入css文件

css in js：利用js的模块化能力去解决样式污染问题，css代码书写起来非常灵活，可以有效的利用js的逻辑，比如函数和变量等。但劣势也同样非常明显，需要在运行时层面上去动态计算每个组件的样式hash，以及动态插入到html文档中，最终造成性能上的损失

而在tailwind面前，其实根本不存在样式污染问题。因为你书写的每一个样式，最终反映到页面上时，都是一个个独立存在的class，本身就是高度原子化和零耦合化，也就不存在任何污染的问题，前提是看语法是否规范

3.4.团队规范的统一 && 省去命名的麻烦

当我们定义好theme配置后，完全可以推广到其它团队一起使用，以tailwind提供的插件功能为基础，去扩展自身的能力，包括添加基础类，以及修改css代码的能力，比如：

      addComponents({
        '.btn': {
          padding: '.5rem 1rem',
          borderRadius: '.25rem',
          fontWeight: '600',
        }
      })

  ]

而平常手写class时，需要额外定义各种类名，使用tailwind可以省去这个麻烦，直接写就完了

4.缺陷

4.1.入门上手问题

对于新手来说，一开始只能边看文档边写css，能把tailwind用好前提是对它定义的class规则很熟悉，这里面是有一定学习成本的

4.2.代码的维护可读性问题

打开一个复杂的页面，也许映入眼帘的是各种乱七八糟的tailwind 定义的class，如果在项目中没能做到有效的样式封装和抽离，在使用tailwind的情况下，会额外增加代码阅读的负担，比如可以在global.css中去封装一个集合类flex-center：

1
2
3
4
5
6

9

@tailwind base;
@tailwind components;
@tailwind utilities;

@layer components {
   .flex-center {
    @apply flex justify-center items-center
  }
}

这样写一个flex的垂直居中就可以直接用flex-center

5.总结

tailwind css提供了一种新的css 解决方案，能够极大地减少css代码的体积，并且解决样式污染问题，回归了最初的css本质，相信未来也会受到更多团队的青睐
