前言

⼀个⻚⾯性能差的话会⼤⼤影响⽤户体验，⽤户打开⻚⾯等待的太久，可能会直接关掉⻚⾯，甚⾄就不再使⽤了，这种情况在移动端更加明显，移动端⽤户对⻚⾯响应延迟容忍度很低

所以我们要了解需要衡量哪些指标，以及如何去做到获取这些指标的数据然后加以分析

1.性能指标

指的是我们在衡量网站性能时所关注的指标

1.1 基本指标

1.1.1. FCP：First Contentful Paint

首次内容绘制时间，即“⽩屏时间“ 可以在js的performance对象中拿到各种信息，最基本的，可以通过performance.getEntriesByType('paint')[1]来获得这个时间

1.1.2. TTI：Time To Interactive

衡量应⽤加载所需时间并能够快速响应⽤户交互的指标 最后⼀个LongTask结束时间，js执⾏时间超过50ms的任务为⻓任务

1.1.3. FID：First Input Delay

从⽤户输⼊到⻚⾯真正开始处理事件回调的潜在最⼤延迟时间 FID 具体的计算⽅法，是从以 FCP 为开始到以 TTI 为结束的这段时间⾥，选择其中js执⾏时间最⻓的任务，⽤它消耗的时间再减去50ms

虽然TTI和FID确实关联良好，但可能存在具有良好TTI分数的⽹站可能具有不良FID分数的情况。例如，站点可能具有⾮常⾼的TTI，但如果⼤多数⽤户在完全加载之前不尝试与站点交互，则这些⽤户将不会遇到⾼TTI的痛苦。 另⼀⽅⾯，站点可能具有相对较低的TTI，但如果站点很早就看起来是交互式的，则⽤户可能会尝试与其进⾏交互，但却发现没有任何反应

1.1.4. TBT：Total Blocking Time

⻚⾯响应⽤户输⼊时，已经被阻塞的时间总和 具体的计算⽅法⽐较明确，统计从FCP到TTI之间的所有⻓任务，并将它们的阻塞部分时间进⾏求和，即为TBT。其中阻塞部分时间指的是⻓任务执⾏时间超过50ms的部分，例如⼀个⻓任务执⾏了70ms，那么阻塞时间就是20ms。可以看出，TBT相⽐于FID是⼀个更加稳定的指标，相较⽽⾔能更加真实地反映⻚⾯对于⽤户输⼊的响应平均延时

1.1.5. FMP：First Meaningful Paint

⾸次有效绘制，表示⻚⾯主⻆元素出现在屏幕上的时间点

Chrome定义的FMP：LayoutObjectsThatHadNeverHadLayout（新增的布局对象）发⽣变化最⼤的下⼀个时刻（paint that follows biggest layout change）

自定义的FMP：⻚⾯加载完毕时间（load） - 开始请求时间（requestStart）；通过MutationObserver给⻚⾯元素设置权重和得分，监听dom加载时间，取权重分最⾼的元素的加载完成时间作为FMP时间

1.1.6. LCP：Largest Contentful Paint

视窗内最⼤的内容元素被渲染的时间 LCP是⼀个⼗分重要并且以⽤户感受为中⼼的指标；它反映了感知层⾯上⻚⾯的加载速度；它标记了⻚⾯主要内容中最⼤内容元素加载完成的时间点；LCP较短的⻚⾯能够让⽤户更快感觉到⻚⾯是可⽤的

根据官⽅⽂档，下列元素会被纳⼊Largest Contentful 元素的考虑范围： 1.<img> 2.<svg>⾥⾯的<image> 3.<video> 4.通过url()函数加载背景图⽚的元素 5.包含text node的块级元素或者inline text的⼦元素

如何确定元素的大小: 1.在viewport内可⻅元素的⼤⼩，如果是超出可视区域或者被裁减、遮挡等，都不算⼊该元素⼤⼩ 2.对于图⽚元素来说，⼤⼩是取图⽚实际⼤⼩和原始⼤⼩的较⼩值，即Min(实际⼤⼩，原始⼤⼩) 3.对于⽂字元素，只取能够覆盖⽂字的最⼩矩形⾯积 4.对所有元素，margin、padding、border等都不算

1.1.7. CLS：Cumulative Layout Shift

累计布局偏移，⻚⾯未加载完引起⻚⾯样式错乱及引起错误操作

Layout Shift Score = Impact Fraction * Distance Fraction，如上：影响分数⼤约为⻚⾯的50%，并将正⽂⽂本向下移动20%。布局移位得分为0.1，0.5 * 0.2的乘积，因此，CLS 为 0.1

1.1.8. SI：Speed Index

衡量⻚⾯可⻅内容填充快慢的指标

计算过程使⽤是开源⼯具speedline通过对⻚⾯进⾏视频录制，并统计⾸帧与最后⼀帧变化的时间差来计算speed Index 的值

1.1.9. TTFB：Time To First Byte（⽤户浏览器接收⻚⾯内容的第⼀个字节所需的时间）

1.1.10. DCL: DOMContentLoaded Event（DOM树完全加载并解析完成的时间）

1.1.11. OL: Onload Event（当整个⻚⾯及所有依赖资源如样式表和图⽚都已完成加载的时间）

1.1.12. 本地资源拦截成功率（从本地缓存加载资源成功量 / 路由解析总量）

2.RAIL模型

RAIL 是一种以用户为中心的性能模型，它提供了一种考虑性能的结构，该模型将用户体验分解到按键操作（例如，点击、滚动、加载）中，帮助我们为每个操作定义性能目标，RAIL 代表 Web 应用生命周期的四个不同方面：响应、动画、空闲和加载。用户对这些上下文分别有不同的性能期望，因此，性能目标是根据上下文以及用户如何感知延迟的用户体验研究来定义的

用户对性能延迟的看法：

1、0~16ms 用户非常关注轨迹运动，他们不喜欢不流畅的动画。如果每秒渲染60个新帧，他们就认为动画很流畅。也就是说，每一帧只有16毫秒时间，这包括浏览器将新帧绘制到屏幕所需的时间，因而应用约有10毫秒的时间来生成一个帧

2、0~100ms 在此时间窗口内响应用户操作会让用户觉得结果是即时呈现的。如果时间更长，操作与用户反应之间的联系就会中断

3、100~1000ms 在此时间窗口内，用户会觉得任务进展基本上是自然连续的。对Web上的大多数用户来说，加载页面或更改视图是一项任务

4、1000ms 或更长超过1000ms(1s)，用户的注意力就会从正在执行的任务上转移

5、10000ms 或更长超过10000ms(10s)，用户会感到失望，并且可能放弃任务。他们以后可能会回来，也可能不会再回来

2.1.响应(response)

目标：在100毫秒内完成由用户输入发起的转换，让用户感觉交互是即时的

时间再⻓，操作与反应之间的连接就会中断，⼈们就会觉得它的操作有延迟。例如：当⽤户点击⼀个按钮，如果100ms内给出响应，那么⽤户就会觉得响应很及时，不会察觉到丝毫延迟感

2.2.动画(animation)

目标：在10毫秒或更短的时间内生成动画的每一帧

从技术上来讲，每帧的最大预算为16毫秒（1000 毫秒 / 每秒60帧 ≈ 16毫秒），但是，浏览器需要大约6毫秒来渲染一帧，因此，准则为每帧10毫秒

实际情况还要根据显示器本身的参数来确定，但一般都是以60帧为标准

2.3.空闲(idle)

目标：最大限度增加空闲时间以提高页面在50毫秒内响应用户输入的几率

空闲周期内运⾏的任务不得超过50ms，当然不⽌RAIL规定，W3C性能⼯作组的Longtasks标准也规定了超过50毫秒的任务属于⻓任务，那么50ms这个数字是怎么得来的呢？

浏览器是单线程的，这意味着同⼀时间主线程只能处理⼀个任务，如果⼀个任务执⾏时间过⻓，浏览器则⽆法执⾏其他任务，⽤户会感觉到浏览器被卡死了，因为他的输⼊得不到任何响应

为了达到100ms内给出响应，将空闲周期执⾏的任务限制为50ms意味着，即使⽤户的输⼊⾏为发⽣在空闲任务刚开始执⾏，浏览器仍有剩余的50ms时间⽤来响应⽤户输⼊，⽽不会产⽣⽤户可察觉的延迟

2.3.加载(load)

目标：在1秒钟内加载⽹⻚并让⽤户看到内容

如果不能在1秒钟内加载⽹⻚并让⽤户看到内容，⽤户的注意⼒就会分散。⽤户会觉得他要做的事情被打断，如果10秒钟还打不开⽹⻚，⽤户会感到失望，会放弃他们想做的事，以后他们或许都不会再回来

3.如何监控

前端性能监控主要分为两种⽅式，⼀种叫做合成监控(Synthetic Monitoring，SYN)，另⼀种是真实⽤户监控(Real User Monitoring，RUM) 合成监控⼀般会借助⼯具来做检查例如：lighthouse。⽽真实⽤户监控的时候，我们就需要采集⼀些⽤户数据

3.1.采集数据

3.1.1.手动采集

为了帮助开发者更好地衡量和改进前端⻚⾯性能，W3C性能⼩组引⼊了Navigation Timing API，实现了⾃动、精准的⻚⾯性能打点；我们可以利⽤ performance API 轻松获取⻚⾯的相关性能数据和衡量⼀些流程的时间

各字段的含义：

  • startTime：有些浏览器实现为navigationStart，代表浏览器开始unload前一个页面文档的开始时间节点。比如我们当前正在浏览baidu.com，在地址栏输入google.com并回车，浏览器的执行动作依次为：unload当前文档（即baidu.com）-> 请求下一文档（即google.com）navigationStart的值便是触发unload当前文档的时间节点

    如果当前文档为空，则navigationStart的值等于fetchStart

  • redirectStart和redirectEnd：如果页面是由redirect而来，则redirectStart和redirectEnd分别代表redirect开始和结束的时间节点
  • unloadEventStart和unloadEventEnd：如果前一个文档和请求的文档是同一个域的，则unloadEventStart和unloadEventEnd分别代表浏览器unload前一个文档的开始和结束时间节点。否则两者都等于0
  • fetchStart是指在浏览器发起任何请求之前的时间值。在fetchStart和domainLookupStart之间，浏览器会检查当前文档的缓存
  • domainLookupStart和domainLookupEnd分别代表DNS查询的开始和结束时间节点。如果浏览器没有进行DNS查询（比如使用了cache），则两者的值都等于fetchStart
  • connectStart和connectEnd分别代表TCP建立连接和连接成功的时间节点。如果浏览器没有进行TCP连接（比如使用持久化连接webscoket），则两者都等于domainLookupEnd
  • secureConnectionStart：可选。如果页面使用HTTPS，它的值是安全连接握手之前的时刻。如果该属性不可用，则返回undefined。如果该属性可用，但没有使用HTTPS，则返回0
  • requestStart代表浏览器发起请求的时间节点，请求的方式可以是请求服务器、缓存、本地资源等
  • responseStart和responseEnd分别代表浏览器收到从服务器端（或缓存、本地资源）响应回的第一个字节和最后一个字节数据的时刻
  • domLoading代表浏览器开始解析html文档的时间节点。我们知道IE浏览器下的document有readyState属性，domLoading的值就等于readyState改变为loading的时间节点
  • domInteractive代表浏览器解析html文档的状态为interactive时的时间节点。domInteractive并非DOMReady，它早于DOMReady触发，代表html文档解析完毕（即dom tree创建完成）但是内嵌资源（比如外链css、js等）还未加载的时间点

  • domContentLoadedEventStart：代表DOMContentLoaded事件触发的时间节点：

    页面文档完全加载并解析完毕之后,会触发DOMContentLoaded事件，HTML文档不会等待样式文件,图片文件,子框架页面的加载(load事件可以用来检测HTML页面是否完全加载完毕(fully-loaded))

  • domContentLoadedEventEnd：代表DOMContentLoaded事件完成的时间节点，此刻用户可以对页面进行操作，也就是jQuery中的domready时间
  • domComplete：html文档完全解析完毕的时间节点

  • loadEventStart和loadEventEnd分别代表onload事件触发和结束的时间节点

计算性能指标：

DNS查询耗时 = domainLookupEnd - domainLookupStart TCP链接耗时 = connectEnd - connectStart request请求耗时 = responseEnd - responseStart 解析dom树耗时 = domComplete - domInteractive 白屏时间 = domloadng - fetchStart domready时间 = domContentLoadedEventEnd - fetchStart onload时间 = loadEventEnd - fetchStart

注意：通过window.performance.timing所获的的⻚⾯渲染所相关的时间点数据，在SPA应⽤中改变了url但不刷新⻚⾯的情况下是不会更新的。因此仅仅通过该api是⽆法获得每⼀个⼦路由所对应的⻚⾯渲染的时间。如果需要上报切换路由情况下每⼀个⼦⻚⾯重新render的时间，需要⾃定义上报
